# 统计分析逻辑文档

## 概述

本文档详细说明了 Nginx WebUI 中统计分析模块的完整逻辑，包括数据采集、处理、存储和查询等各个环节。

## 1. 分析任务架构

### 1.1 时间范围递进策略

分析任务按照时间范围递进执行，统一为一个分析任务：

- **5分钟** (`5/60` 小时) - 实时监控
- **1小时** (`1` 小时) - 短期趋势
- **1天** (`24` 小时) - 中期趋势
- **7天** (`168` 小时) - 长期趋势

### 1.2 触发机制

1. **日志监听触发** (`watcher`)
   - 使用 `pyinotify` 监听访问日志文件变化
   - 日志文件修改时触发分析（延迟30秒合并触发）
   - 自动按时间范围递进分析

2. **兜底定时触发** (`auto_fallback`)
   - 每5分钟执行一次
   - 防止监听失效或长时间无请求时仍能刷新统计
   - 自动按时间范围递进分析

3. **手动触发** (`manual` / `manual_full`)
   - 用户通过前端界面手动触发
   - 支持增量分析（`full=false`）和全量分析（`full=true`）
   - 自动按时间范围递进分析

## 2. 分析模式

### 2.1 增量分析（默认）

**特点：**
- 使用 `pygtail` 从上次读取位置继续读取新日志
- 基于"上次分析时间"作为锚点，只分析新日志
- 合并历史缓存数据，避免重复计算
- 适合日常运行，性能高效

**流程：**
1. 读取历史缓存，获取 `previous_last_analysis_time`
2. 使用 `pygtail` 从上次位置读取新日志
3. 解析日志行，过滤时间范围
4. 统计新数据并与历史数据合并
5. 按小时保存到数据库

### 2.2 全量分析

**特点：**
- 从文件尾部开始读取整个日志文件
- 不进行时间过滤，分析所有日志
- 使用分批处理（每批1万条），避免内存问题
- 适合日志结构变化或异常后手动重算

**流程：**
1. 从文件尾部开始读取（从新到旧）
2. 分批处理，每批1万条
3. 每批处理完后立即统计并合并结果
4. 按小时保存到数据库
5. 处理完一批后清空批次，释放内存

## 3. 数据保存机制

### 3.1 按小时保存策略

无论是增量分析还是全量分析，都采用**按小时保存**的策略：

1. **跨小时检测**
   - 处理每条日志时，检查其所属小时
   - 当检测到跨小时时，立即保存上一小时的数据

2. **保存时机**
   - **增量分析**：跨小时时保存 + 分析完成后保存最后一小时
   - **全量分析**：跨小时时保存 + 处理完成后保存最后一小时

3. **数据合并**
   - 保存时合并历史缓存数据
   - 确保数据完整性和准确性

### 3.2 保存的数据结构

每次保存包含以下数据：

## 4. 日志解析

### 4.1 解析方式

使用 `pygrok` 库解析 Nginx 访问日志，支持标准 combined 格式：

```
$remote_addr - $remote_user [$time_local] "$request" $status $body_bytes_sent "$http_referer" "$http_user_agent"
```

**解析字段：**
- IP 地址
- 用户（通常为 `-`）
- 时间戳
- HTTP 请求（方法、路径、协议）
- 状态码
- 响应大小
- Referer
- User-Agent

### 4.2 时间桶统计

根据时间范围自动选择时间桶粒度：

- **5分钟及以下**：按5分钟对齐（00, 05, 10, ..., 55）
- **1小时到1天**：按小时聚合
- **1天及以上**：按天聚合

## 5. 攻击检测

### 5.1 检测规则

1. **状态码检测**
   - 4xx/5xx 状态码可能表示攻击尝试

2. **路径检测**
   - 检测常见攻击路径（如 `/admin`, `/wp-admin`, `/phpmyadmin` 等）
   - 检测路径中的攻击模式（SQL注入、XSS等）

3. **User-Agent 检测**
   - 检测可疑的 User-Agent

4. **Referer 检测**
   - 检测可疑的 Referer

### 5.2 攻击记录

检测到的攻击会记录详细信息，包括：
- 时间戳
- IP 地址
- 访问路径
- HTTP 状态码
- 攻击类型

最多保留最近50条攻击记录。

## 6. 数据查询接口

### 6.1 接口列表

所有查询接口都从缓存读取数据，不进行实时分析：

1. **`GET /api/statistics/summary`** - 获取基础统计数据
   - 返回总数、成功、错误、错误率等
   - 包含分析任务状态信息

2. **`GET /api/statistics/trend`** - 获取时间趋势数据
   - 返回时间趋势图数据

3. **`GET /api/statistics/top-ips`** - 获取 Top IPs
   - 返回访问量前N的IP地址

4. **`GET /api/statistics/top-paths`** - 获取 Top Paths
   - 返回访问量前N的路径

5. **`GET /api/statistics/status-distribution`** - 获取状态码分布
   - 返回各状态码的分布情况

6. **`GET /api/statistics/attacks`** - 获取攻击检测记录
   - 返回攻击检测记录列表

7. **`GET /api/statistics/overview`** - 获取统计概览
   - 返回完整的统计数据

8. **`GET /api/statistics/realtime`** - 获取实时统计数据
   - 返回最近1小时的实时数据

### 6.2 查询参数

所有接口都支持 `hours` 参数，指定查询的时间范围：
- 默认：24小时
- 范围：1-168小时（7天）

## 7. 性能优化

### 7.1 内存优化

1. **分批处理**
   - 全量分析时，每批处理1万条日志
   - 处理完一批后立即释放内存

2. **流式读取**
   - 使用缓冲区从文件尾部读取
   - 避免一次性加载整个文件

3. **及时释放**
   - 处理完的数据立即释放
   - 避免内存累积

### 7.2 多线程优化

1. **并行统计**
   - 使用 `ThreadPoolExecutor` 并行处理日志条目
   - 数据量大于1000条时启用多线程

2. **线程池大小**
   - 最大工作线程数：`min(4, cpu_count())`
   - 根据数据量自动调整

### 7.3 缓存优化

1. **缓存策略**
   - 数据按小时保存到数据库
   - 查询时直接从缓存读取

2. **缓存清理**
   - 自动清理超过30天的缓存数据
   - 防止数据库膨胀

## 8. 错误处理

### 8.1 异常处理

1. **分析异常**
   - 捕获所有异常，记录错误日志
   - 确保分析状态正确更新
   - 不影响后续分析任务

2. **保存异常**
   - 保存失败不影响分析结果
   - 记录警告日志
   - 下次分析时会重新保存

### 8.2 状态管理

使用全局状态 `ANALYSIS_STATE` 跟踪分析任务：

- `is_running`: 是否有分析任务在执行
- `last_start_time`: 最近一次分析开始时间
- `last_end_time`: 最近一次分析结束时间
- `last_error`: 最近一次错误信息
- `last_success`: 最近一次是否成功
- `last_trigger`: 最近一次触发方式
- `last_duration_seconds`: 最近一次任务耗时
- `watcher_enabled`: 是否启用了日志监听

## 9. 数据清理

### 9.1 自动清理

- **清理策略**：超过30天的缓存数据自动清理
- **清理时机**：每次分析任务完成后执行
- **清理范围**：所有过期的统计缓存记录

### 9.2 手动清理

可以通过调用 `cleanup_old_cache(max_age_hours=30*24)` 手动清理。

## 10. 日志记录

### 10.1 日志级别

- **INFO**: 分析开始、完成、保存等关键操作
- **WARNING**: 保存失败、清理失败等非致命错误
- **ERROR**: 分析异常、严重错误
- **DEBUG**: 详细的调试信息

### 10.2 关键日志

- 分析任务开始/结束
- 每小时数据保存
- 分析进度（每批处理完成）
- 错误和异常信息

## 11. 配置说明

### 11.1 时间范围配置

分析任务按以下时间范围递进：
```python
time_ranges = [
    (5 / 60, "5分钟"),  # 5分钟 = 5/60 小时
    (1, "1小时"),
    (24, "1天"),
]
```

### 11.2 批次大小配置

- **全量分析批次大小**：`BATCH_SIZE = 10000`（1万条）
- **增量分析**：数据量通常较小，不进行分批

### 11.3 缓存配置

- **缓存键生成**：基于时间范围和当前小时
- **缓存有效期**：查询接口根据需求设置（5分钟到无限制）

## 12. 数据流程

```
日志文件变化/定时触发
    ↓
按时间范围递进分析（5分钟 → 1小时 → 1天）
    ↓
读取日志（增量/全量）
    ↓
解析日志行
    ↓
统计处理（IP、状态码、路径、方法、攻击检测）
    ↓
按小时保存到数据库
    ↓
前端查询接口从缓存读取
```

## 13. 注意事项

1. **数据一致性**
   - 保存时合并历史数据，确保数据完整
   - 跨小时保存时，确保数据不丢失

2. **性能考虑**
   - 大文件全量分析时，使用分批处理
   - 及时释放内存，避免内存溢出

3. **并发控制**
   - 使用任务计数器管理并发任务
   - 防止重复触发分析任务

4. **错误恢复**
   - 分析失败时，状态正确更新
   - 下次分析时会自动恢复
