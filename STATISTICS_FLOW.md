# 统计分析流程文档

## 核心设计理念

全量分析和增量分析的**逻辑完全一样**，只是数据源不同：
- **增量分析**：使用 pygtail 读取新增的日志行
- **全量分析**：从文件尾部读取整个文件（限制最近7天）

## 完整数据流

```

                            ↓
┌─────────────────────────────────────────────────────────────┐

                            ↓
┌─────────────────────────────────────────────────────────────┐

│  - 时间粒度：小时                                                │
```

## 数据存储结构

### StatisticsCache5Min 表（5分钟数据）
- **cache_key**: `5min_2025-12-02_16`（小时级别）
- **data**: JSON格式，包含：
  - `hourly_trend`: 5分钟粒度的时间桶数据
  - `summary`: 汇总统计（总请求数、错误率等）
  - `top_ips`, `top_paths`: Top数据
  - `analyzed_lines`: 本次任务实际处理的行数
- **保留时长**: 7天

### StatisticsCache 表（1小时、1天数据）
- **1小时数据**:
  - `cache_key`: `1_2025-12-02_16`
  - `hourly_trend`: 小时粒度
  - 保留时长：7天

- **1天数据**:
  - `cache_key`: `24_2025-12-02`
  - `hourly_trend`: 天粒度
  - 保留时长：7天

## 关键参数说明

### analyzed_lines（分析行数）
- **含义**: 本次任务实际处理的日志行数
- **计算**: `in_range_count`（时间范围内的新日志行数）
- **不是**: 累积的总请求数
- **用途**: 显示任务处理进度和工作量

### 全量分析范围
- **统一范围**: 最近7天
- **原因**: 超过7天的数据不再分析，减少处理时间
- **数据保留**: 7天后自动清理

### 时间粒度映射
| 查询范围 | 数据来源 | 时间粒度 | 数据点数量（约） |
|---------|---------|---------|----------------|
| 1小时   | StatisticsCache5Min | 5分钟 | 12个 |
| 24小时  | StatisticsCache (24h) | 小时 | 24个 |
| 7天     | StatisticsCache (24h) | 天 | 7个 |

## 任务状态管理

### AnalysisStateManager
统一的状态管理器，确保状态一致性：

```python
_state_manager.start_task(trigger="manual")    # 开始任务
_state_manager.update_progress(time, lines)    # 更新进度
_state_manager.finish_task(success, error, analyzed_lines)  # 完成任务
_state_manager.reset()                          # 重置状态（重启时）
```

### 状态字段
- `is_running`: 是否有任务运行中
- `task_counter`: 当前运行的任务数
- `last_analyzed_lines`: 最后一次任务处理的行数
- `current_analyzed_lines`: 当前任务实时处理的行数
- `last_start_time`, `last_end_time`: 任务开始和结束时间
- `last_duration_seconds`: 任务耗时
- `watcher_enabled`: 是否启用文件监听

## 任务类型

### 增量分析（增量按钮）
- 任务数：2个（1小时聚合 + 1天聚合）
- 5分钟任务：跳过（使用现有缓存）
- 耗时：秒级

### 全量分析（全量按钮）
- 任务数：3个（5分钟读取 + 1小时聚合 + 1天聚合）
- 5分钟任务：读取最近7天的日志文件
- 耗时：2-3分钟（377MB日志文件）
- 进度：每5000行输出一次

## 自动任务

### 文件监听（inotify）
- 检测 access.log 变化
- 延迟30秒触发（防抖）
- 执行5分钟 → 1小时 → 1天分析

### 兜底定时任务
- 每5分钟执行一次
- 重启后等待5分钟再开始第一次分析
- 执行5分钟 → 1小时 → 1天分析

## 缓存清理

### 清理策略
- **清理间隔**: 每次分析任务后清理
- **保留时长**: 7天
- **清理范围**: StatisticsCache5Min 和 StatisticsCache 两个表

### 清理时机
- 文件监听触发分析后
- 兜底定时任务执行后
- 手动触发分析后

## 性能优化

### 日志读取
- **大文件处理**: 377MB文件，约1600行/秒
- **流式读取**: 避免一次性加载到内存
- **批次处理**: 全量100000行/批，增量1000行/批

### 进度反馈
- **状态更新**: 每行日志处理后更新 `current_analyzed_lines`
- **日志输出**: 每5000行输出一次进度
- **文件读取进度**: 每100MB输出一次

### 并行统计
- 使用线程池并行处理统计计算
- 避免阻塞主线程
- 独立任务和任务组统一管理

## API 接口

### 查询接口
- `GET /api/statistics/summary?hours=1` - 汇总数据
- `GET /api/statistics/trend?hours=1` - 趋势数据
- `GET /api/statistics/top-ips?hours=24` - Top IP
- `GET /api/statistics/top-paths?hours=24` - Top 路径
- `GET /api/statistics/task-status` - 任务状态

### 操作接口
- `POST /api/statistics/analyze?full=false` - 增量分析
- `POST /api/statistics/analyze?full=true` - 全量分析

## 数据闭环验证

1. ✓ 5分钟数据存储到数据库（StatisticsCache5Min）
2. ✓ 1小时数据从5分钟缓存聚合，存储到数据库（StatisticsCache）
3. ✓ 1天数据从1小时缓存聚合，存储到数据库（StatisticsCache）
4. ✓ 页面查询从数据库读取，不重新分析
5. ✓ 7天后自动清理过期数据
6. ✓ analyzed_lines 正确显示单次任务处理量

## 重启后行为

1. ✓ 状态自动重置（is_running=False）
2. ✓ 清理所有任务组
3. ✓ 不立即启动分析
4. ✓ 等待5分钟后开始第一次兜底分析
5. ✓ 文件监听正常启动（如果支持）
